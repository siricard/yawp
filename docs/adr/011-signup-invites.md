# ADR 011 — Signup via invite link

**Status:** Accepted (Phase 1 design)

**Date:** 2026-05-18

## Context

A Yawp identity is created the moment a client generates a keypair — it does not depend on a server. But becoming usable requires picking at least one anchor (ADR 006). The signup question is therefore **not** "how do we provision an identity?" but **"how does a new user gain admission to an anchor?"** Open-signup anchors are the obvious choice but invite spam, generate moderation burden, and contradict the friend-group framing (ADR 010). For v1 we constrain the signup surface deliberately.

## Decision

### Pattern A: invite-link required

For v1, every anchor signup goes through an **invite link** issued by the anchor's operator. There is no open-signup form on the public web UI.

The invite link encodes:

- The anchor hostname (e.g., `a1.example.com`).
- A signed invite token (one-time or shareable; see flavors below).
- An optional expiry timestamp.

When a new client opens the link, it:

1. Generates the user's keypair locally (if not already created).
2. Submits the public key + signed proof-of-key-possession + invite token to the anchor.
3. The anchor verifies the invite, accepts the user, and the anchor becomes the user's first anchor.

### Two invite flavors

**Single-use, time-limited**
- Generated by an operator for a specific recipient.
- Valid for one signup only.
- Expires after a configurable window (default: 7 days).
- Suitable for "send Alice a link."

**Shareable**
- Generated by an operator (or, optionally, by an existing user with operator-granted permission).
- Anyone with the link may sign up.
- Has an optional expiry.
- Suitable for "post the link in our Discord and let people self-onboard."

Both flavors are revokable by the operator at any time. Revocation invalidates all in-flight signups using that link.

### Multi-anchor: add more later

The invite admits the user to **one** anchor. Adding additional anchors later is a separate flow:

- The user, already authenticated to their current anchor(s), requests anchoring at a new server.
- If that server requires an invite, the user must obtain one through the same invite-link mechanism.
- If the server is configured for open re-anchoring (rare), the user can self-add.

There is no "transfer" — the DID does not change. The user simply gains a new anchor and updates their PPE to include it.

### What the invite *doesn't* do

- It does not assign a DID. The DID is `base58(sha256(public_key))` and the client computes it.
- It does not assign a username or handle. Display name is in the PPE; anchor handles (`alice@a1.example.com`) are issued by the anchor independently.
- It does not unlock private settings or recovery — those are mnemonic-derived (ADR 007).

### Onboarding flow summary

1. User clicks invite link (or scans invite QR).
2. Client generates keypair, displays mnemonic, asks user to save it.
3. Client submits public key + invite token to the anchor.
4. Anchor creates the user record, stores the PPE and (empty) private blob.
5. Client establishes its first session (ADR 012) and prompts the user to add a second anchor "for safety" (deferred prompt after first week — see grilling note in CONTEXT.md).

### Operator UI for invites

The anchor operator's admin UI provides:

- A "create invite" form: single-use vs shareable, expiry, optional note for tracking who it was given to.
- A list of active invites with redemption status.
- A revoke action.

## Consequences

### Positive

- Friend-group anchors stay friend-group-sized. The operator controls the door.
- Public-anchor operators can still scale via shareable links + Tier 2 budgets (ADR 010), without needing to wire up a separate registration flow.
- Spam signup is structurally prevented — no anonymous self-service signup endpoint exists in the wire format.
- The same primitive serves all signup paths (whether the operator hand-picks each user or posts a shareable link on a forum).

### Negative

- An operator who wants truly open signup has to issue a perpetually-valid shareable link and accept the abuse risk. Tier 2 budgets help but don't eliminate it.
- New users who don't already know an operator have no path in. Yawp will need a list of "anchors accepting new users" somewhere (out of scope for the protocol).
- Operator burden: generating invites is a manual step. UX must make it fast (1-click + share).

### Rejected alternatives

- **Open self-service signup on every anchor.** Rejected for v1: would invite immediate spam abuse and conflicts with the friend-group framing.
- **Federated signup (any anchor can vouch for you on any other anchor).** Rejected for v1: opens a credential delegation channel we don't want until we have a clear story for revocation.
- **Email-verified open signup.** Rejected: introduces a server-side identity attribute (email) we deliberately don't want; doesn't actually stop motivated abuse.
- **CAPTCHA-gated open signup.** Rejected: doesn't actually stop motivated abuse and adds operator dependency on third-party CAPTCHA providers.
