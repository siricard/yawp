## Identity

**User**
A person, identified globally by a single DID. Yawp users are not server-bound — the DID is portable. The user is _the same user_ across every server they interact with.

**DID**
The user's canonical identifier: `did:yawp:<base58(sha256(public_key))>`. Derived from their Ed25519 public key. No server controls it; no server can issue one to someone else.

**Display name**
The free-form human-readable name the user has chosen for themselves. Not unique. Lives in the PPE. Travels with the DID. **Default at signup**: a friendly random word-pair-and-number (e.g., `silver-fox-42`) generated by the client when the user skips the name prompt in step 7 of the invite-redemption flow. Settable later from Settings. The default is purely cosmetic — the DID is the only identity-bearing value.

**Anchor handle**
An optional short alias issued by a specific anchor server (`alice@a1.example.com`). Convenience alias for WebFinger-style lookup. The handle is _anchor-issued_, not user-owned — different anchors may give the same user different handles. Not synced between anchors. **The DID is identity. The handle is a lookup convenience.**

**Discovery URL**
A shareable URL that points to one anchor where the user's PPE can be fetched (`yawp://a1.example.com/u/<did>`). Used for cold contact (QR codes, share links, deep links). Once a peer has fetched the PPE, they know all the user's anchors and don't need the discovery URL again.

## Peers & verification

**Peer**
The user-facing term for "another Yawp user this user has interacted with or can DM." Peers appear on the dedicated DM page in Pinned / Recent / All groupings; their entries show display name, avatar, key fingerprint, and verification status. Replaces the earlier "contact" working-term. A peer relationship is **implicit** — derived from DM history and shared workspaces — not an explicit add-friend-request action. There is no separate "friends" concept in v1.

**Key fingerprint**
A short human-displayable rendering of a peer's DID public key. Used in the UI for **out-of-band verification**: two users compare fingerprints in person, on a video call, or via a trusted channel, then mark the peer as verified in the app. Fingerprints are derived deterministically from the public key — same key always produces the same fingerprint. **Format**: the first **128 bits** of `sha256(master_public_key)`, rendered as `yp:` followed by 4 groups of 4 hex characters separated by `·` (e.g., `yp:8f3a · d21c · 47ee · 0b91`). The shorter forms used in some design mocks (e.g., `yp:8f3a · d21c`) are visual shorthand only — production UI uses the full 128-bit form.

**Verification status**
Per-peer state recording whether the local user has manually confirmed that peer's key fingerprint via out-of-band verification. Three states:

- `unverified` (default): peer's key is trusted-on-first-use; no manual confirmation yet. UI may show a "pending verify" chip.
- `verified`: user has confirmed the fingerprint out-of-band and marked the peer verified. UI shows a shield-check tick. Carries a `verified_at` timestamp.
- `key_changed`: peer's **master key** has changed since last verification (master-key rotation, mnemonic-based recovery to a new key, or impersonation). Surfaced loudly in the UI — user must re-verify. New device subkeys do **not** trigger `key_changed`: the fingerprint covers the master public key only, so routine subkey churn (new phone, reinstall) is invisible at this layer by design — avoiding alarm fatigue.

Verification status lives in the user's private blob (per-peer record); synced across the user's devices. Not visible to peers or to servers.

**Verification UX boundaries**
Verification is **non-gating** — communication with unverified peers is unrestricted; DMs flow without prompts, banners, or interrupts. Verification surfaces appear in exactly three places:

1. **Peer profile sheet** — fingerprint shown in muted text alongside an "Verify identity" affordance (opens a modal offering QR-scan in-person or fingerprint-compare out-of-band).
2. **Security settings** — global list of verified peers with revoke/re-verify controls.
3. **`key_changed` banner** — appears at the top of an affected DM thread **only** when a previously-`verified` peer's key changes. Dismissable; non-blocking. Unverified peers receive no banner on key change (the user already accepted TOFU).
   The chat composition and message rendering surfaces themselves are free of verification UI. The goal is **zero friction in the seamless path**.

**Pinned peer**
A peer the local user has explicitly elevated above the chronological "Recent" list on the DM page. Per-user, stored in the private blob. Cosmetic — does not affect notification policy, federation, or anything outside the DM page's grouping.

## Server roles

**Server**
A Yawp instance (single deployed Phoenix node, single Docker compose stack). Servers host rooms and accept user submissions. Servers do _not_ hold user identities — they hold user _relationships_. The user-facing term for a server in the client UI is **workspace** (see below).

**Workspace** (UI term)
The user-facing label for a server in the client UI. Functionally identical to "server" — workspace and server refer to the same protocol entity. Documents and protocol discussions use "server"; UI strings and product copy use "workspace."

**Anchor server**
A server that a user has chosen to hold their canonical user data (PPE and private blob). A user may have multiple anchor servers; they replicate the user's data among themselves. The anchor concept is **opt-in by the user**, not a property of the server itself — any server is willing to be someone's anchor if the user requests it.

**Guest server**
A server the user is _present_ on but has not chosen as an anchor. The guest server knows the user's DID and caches their PPE (so it can render display name + avatar), but it does not store the private blob. Most servers, for most users, are guest servers.

The same server can be an anchor for some users and a guest for others. "Anchor" and "guest" describe the _relationship between a user and a server_, not the server's nature.

## User data

**PPE (Public Profile Envelope)**
A signed bundle of the user's public-facing data. Stored plaintext on every server that knows the user (whether anchor or guest); integrity is enforced by the signature. Fields include: display name, avatar reference, bio, list of anchor servers, public key, profile_version, and device subkeys with opaque device IDs.

The PPE is **signed by the user's master key**. Any tampered copy is rejected on verification. Encryption at rest is not part of the protocol — it's the server operator's responsibility (disk encryption, Postgres TDE).

**Private settings blob**
A bundle of the user's private data, encrypted client-side with a key only the user knows. Anchor servers store the ciphertext and replicate it but cannot read it. Contains: room memberships, contact list, DM history pointers, blocked users, mute list, notification preferences, device names/platforms/push tokens, and optional encrypted private-key backup.

**profile_version**
A monotonically increasing version number on the PPE, signed alongside it. Cache invalidation key: when a server sees a higher version in any incoming message envelope, it knows its cached PPE is stale and refetches.

**Device subkey**
An Ed25519 subkey for one of the user's devices, signed by the user's master key. Lives in the PPE alongside an opaque device ID. Used so peers can E2EE-encrypt messages to _each_ device. Per-device revocation: revoking one subkey leaves others intact.

**Device metadata**
Human-readable information about each device (name, platform, last-seen). Lives in the private blob — only the user's own clients can read it. Push tokens are a separate concern (see _Device push registry_ below).

**Device push registry**
A per-user, anchor-only, plaintext (to the anchor) registry mapping each device subkey ID to its current OS push token (APNs or FCM). Held at the user's anchors, **not** in the encrypted private blob, because the anchor must be able to read the token to fire mobile pushes through APNs / FCM. Not part of the PPE — never replicated to guest servers. A privacy trade-off acknowledged in: the user's anchors necessarily learn _when_ a push is fired, even if not _what_ it contains.

## Messaging & routing

**Direct Message (DM)**
A message addressed to one or more specific DIDs. Routed through the sender's anchor server, which delivers to the recipient's anchor servers, which dispatch to the recipient's devices. 1:1 and group DMs share the same envelope shape — the only difference is the size of the recipient list.

**DM envelope**
The signed wire shape for a DM. Carries `envelope_id`, `sender_did`, `recipient_dids` (one or more), `timestamp`, `body` (plaintext in v1; ciphertext in M10+), `attachments`, `reply_to`, `mentions`, and a `sender_signature` produced by the sender's device subkey over the canonical-JSON of the envelope body. The delivery wrapper signs and stamps this envelope when crossing federation boundaries.

**Conversation ID (DM)**
The deterministic identifier for a DM thread: `sha256(sorted set of recipient DIDs ∪ {sender_did})`. Same set of people always produces the same conversation ID — no server needs to allocate one. Works identically for 1:1 and group DMs.

**Group DM**
A DM with two or more recipients. The roster is **immutable for the conversation's life** — adding or removing a participant means starting a new conversation (with a new conversation ID). v1 ships group DMs but defers mutable group-DM roster management.

**Delivery state**
Per-recipient lifecycle state for a DM, tracked end-to-end. Three states: `sent` (sender's anchor accepted the envelope), `delivered` (recipient's anchor wrote the envelope to its inbox; signed `delivery_ack` returned), `read` (recipient's device updated its read marker for the conversation; propagated through both anchors). Group DMs surface aggregate state ("delivered to 2/3, read by 1/3").

**Read receipt**
The propagated signal that a recipient's read marker for a DM conversation has advanced. Senders may globally **opt out of sending read receipts** via a private-blob preference mirrored to their anchor, which then declines to forward outbound read-marker updates. Per-conversation opt-out is deferred.

**Room**
A persistent multi-user conversation hosted on exactly one server. The server hosting the room is the room's _host server_ for everyone in the room. Room membership lives in the host server's database. A room is **single-host** — only one server is authoritative for it at any moment.

**Host server (of a room)**
The single server authoritative for a room's message log, membership, settings, and moderation state. The room's host server is the entry point for all room writes.

**Room message**
A message addressed to a room. Submitted by the sender's client directly to the room's host server. Not relayed through the sender's anchor.

**Room migration**
The operator-driven flow that moves a room from one host server to another. The source host exports the message log + membership + settings; the destination host imports it; participants' clients are notified via a signed migration event in the room timeline and update their host pointer. The DID of every member is unchanged. Used when an operator winds down a server, switches VPS, or moves to managed hosting.

**Room roster**
The authoritative member list of a room, owned by the room's host server. Source of truth for "who is in this room." Used to enforce bans, kicks, and per-room permissions.

**Server owner**
The single user who owns the server (typically the operator). Has unconditional authority over the server: roles, bans, room creation/deletion, server settings. There is no "room owner" — authority over rooms derives from server-level permissions.

**Role (server-scoped)**
A named bundle of permission bits attached to a user's server membership. Roles are defined and assigned at the **server level**; their effects apply across all rooms on that server unless overridden per channel. Roles are not portable across servers.

**System role**
A role created automatically by the system when a server is provisioned, marked so that users cannot delete it. v1 seeds three system roles per server: `Owner`, `Admin`, `Member`. They share the same data shape as user-defined roles — the distinction is purely a `system: true` flag that suppresses delete actions in the UI. User-defined ("custom") roles are supported in the data model but not exposed in the v1 frontend.

**Channel override**
A per-room (per-channel) adjustment to the permissions a role or user has in that specific room. Used to grant access to private rooms, restrict access to sensitive rooms, or fine-tune posting/management rights inside a single channel. Channel overrides modify the server-wide baseline; they do not replace it.

**Private-channel invite**
The act of granting a specific user a channel override that lets them see and participate in a private room they could not otherwise access. Implemented as a permission override on the channel, not as a separate room-roster concept. Revokable by anyone with sufficient role permissions.

**Room list (cached)**
The client-side view of which rooms a user belongs to, cached in the user's private blob and synced through anchors. Convenience for fast app launch; the host roster is authoritative if the two disagree.

**Room presence**
Real-time online/active state of users within a specific room. Managed by the room's host server via Phoenix Presence — not propagated through anchors. Distinct from the user's coarse global presence (online / idle / offline) which is brokered by the user's anchor.

**Room visibility**
A per-room setting controlling who can _see_ the room exists. Two values: `private` (not listed; only known via invite or explicit permission grant) and `server-public` (visible to **every guest of the host server**, Discord-style — no per-room admission needed). Federation-wide discovery is not in scope for v1.

**Channel type**
A room's interaction mode. Two values in v1: `text` (persistent chat) and `voice` (persistent WebRTC voice room — successor to M5's ephemeral 1:1 calls). Both share the same membership / permission / role / override model.

**SFU (Selective Forwarding Unit)**
The in-process media topology used by voice channels. Each participant sends one upstream RTP stream to the host server's SFU; the SFU forwards each stream to each other participant individually. Built on `ex_webrtc` and runs inside the Phoenix app under OTP supervision. Voice channels are SFU-mediated from v1 onward — no peer-mesh fallback.

**TURN relay**
A media-relay service (typically `coturn`) used by WebRTC clients when direct peer-to-peer connectivity fails — most commonly when participants sit behind symmetric NAT (CGNAT on cellular, corporate firewalls, double-NAT home setups). Yawp ships `coturn` as a sidecar service in docker-compose by default. TURN is **required infrastructure** for voice to work reliably on cellular networks. The data plane never crosses Yawp itself — clients connect directly to the TURN relay.

**TURN REST credentials**
Short-lived per-session TURN credentials issued by Yawp using a shared HMAC secret pre-configured into both Yawp and coturn. Yawp computes ephemeral `(username, password)` pairs at the start of each voice session; coturn validates them against the shared secret with no per-session call to Yawp.

**Voice channel participant cap**
The maximum number of simultaneous participants in a single voice channel. Default `25` in v1; configurable per server by self-hosting operators based on their hardware. Exceeded join attempts are refused with a "channel full" response.

**Category**
A named grouping of channels within a server. One level of hierarchy in v1 (a channel belongs to at most one category, a category has no parent). The data model uses a self-referential `parent_id` so deeper nesting can be enabled later without migration; v1 enforces depth ≤ 1 at the API layer. Categories are UI organization only — they carry no permissions in v1 (permission cascade from categories is a Tier 3 RBAC feature).

**Edit mode**
A client-side toggle (admins+ only) that enables destructive UI actions: drag-to-reorder channels and categories, drag-to-recategorize, delete shortcuts. Off by default to prevent accidental fat-finger reordering during normal navigation. Pure UX, not a server-side state.

## Client navigation shell

**Workspace bar** (web)
The horizontal strip at the top of the desktop UI listing the user's joined workspaces as small badge tiles. Active workspace is highlighted. A dedicated `@` (DM) tile sits alongside the workspaces. Order is user-controlled, drag-reorderable; the order is stored in the user's private blob and syncs across devices.

**Tab row** (web)
The horizontal row directly below the workspace bar. Mixes two sections:

1. A **Recent DM strip** — chips of the user's most-active DM peers; clickable to open the DM in place. A "More →" affordance opens the dedicated DM page.
2. The **active workspace's channels** — text and voice channels of the currently-selected workspace, grouped by category.
   Allows DMs and the current workspace's channels to coexist in one nav so the user never has to leave workspace context to glance at DMs.

**Voice dock**
A small persistent widget anchored in the top bar whenever the user is in a voice channel. Shows the room name, stacked avatars of current participants, and mic/deafen/leave controls. Stays visible regardless of which workspace / channel / DM the user is viewing — voice is global to the client.

**Mobile tab bar**
The three-tab bottom navigation in the mobile client:

- **Spaces** — workspace browser; functionally equivalent to the desktop workspace bar + tab row.
- **Direct** — dedicated DM destination (the peer list with Pinned / Recent / All).
- **You** — identity, settings, and the user's own fingerprint / QR code.

**Room join policy**
A per-room setting controlling who can _join_ once they can see the room. Two values: `invite-only` (must redeem a valid invite) and `open` (anyone who can see the room may join without invite).

**Room invite (warm)**
A signed invite delivered as a structured DM from an existing member to a recipient the inviter can already DM (shared room, accepted contact). Recipient's client renders a "Join room" affordance. Requires no separate cross-server credential — it rides on the existing DM federation path.

**Room invite link (cold)**
A shareable URL (`yawp://<host>/r/<id>?token=<...>`) redeemable by anyone who has the link. Pasted into the app's "Join via link" entry point. Used when the inviter and invitee cannot DM each other (no shared server, no prior contact). Redeeming the link auto-promotes the user to **guest** on the host server (granting visibility into all server-public rooms) and then joins them to the specific room the link refers to. Invite links can be single-use or shareable, time-limited or open-ended, and are revokable by room admins.

**Auto-guest on join**
When a user joins a room on a server they have no prior relationship with, the host server creates a **guest** session for them (not an anchor relationship). The user becomes "present" on the server — the server caches their PPE and accepts their writes to the server's rooms — but the user's canonical data stays at their anchors elsewhere. Leaving every room on a server returns that server to having no relationship with the user.

Becoming a guest on a server grants visibility into **all server-public rooms on that server** (Discord-style), not just the room that triggered the join. Private rooms remain gated by invite or per-room permission. The unit of admission is the server, not the individual room.

**Inbox**
The user's pending-delivery queue, held by their anchor servers. When a peer DMs a user who has no connected devices, the anchor holds the encrypted envelope until a device connects and picks it up.

## Sessions & connection model

**Session**
An authenticated relationship between one user device and one server. Created via challenge-response (see) and represented by a server-issued session token. Sessions are per-server: a user with 50 servers has 50 sessions.

**Refresh token**
A long-lived (e.g. 30-day) opaque token issued by a server after successful challenge-response. Lets a device get a new session token without re-signing a challenge on every app launch.

**Always-on connection**
A persistent websocket maintained between a user's client and one of their anchor servers. Used for real-time delivery of DMs, notifications, and presence updates. The anchor must be reachable to maintain this.

**On-demand connection**
A websocket opened to a guest server when the user is actively viewing or interacting with that server (open the app to a room → open the connection). Closed when the user moves elsewhere.

**Presence broker**
Function performed by an anchor server: track its user's online state and propagate changes to subscribed guest servers. Lets the user appear online on all 50 of their servers without each server needing a direct connection.

**Degraded mode**
The state a client enters when none of the user's anchor servers are reachable. Falls back to direct guest-server connections. Presence appears offline globally; DMs cannot be delivered until at least one anchor comes back. UI shows a prominent warning.

## Federation primitives

**Server keypair**
An Ed25519 keypair owned by each Yawp instance (server), distinct from any user's keypair. Used to sign server-to-server federation envelopes. Published at `https://<host>/.well-known/yawp/server-key.json` as a list of keys with `not_before` / `not_after` validity windows.

**Delivery wrapper**
The signed envelope produced by a sending anchor when forwarding a DM to a recipient's anchor. Contains the user-signed message envelope, a delivery nonce, a timestamp, the sender anchor identifier, and a signature by the sender anchor's server keypair. The recipient anchor verifies both the user signature on the inner envelope and the server signature on the wrapper.

**Key document**
The JSON document at `/.well-known/yawp/server-key.json` listing a server's current and historical keys, their validity windows, and any revoked key IDs. Peers cache this document with a default TTL of 24h.

**Federation API**
HTTPS endpoints exposed by each anchor for cross-operator anchor-to-anchor communication (PPE replication, inbox push, presence subscribe, etc.). Versioned wire format with signed payloads. Independent of BEAM distribution, which is reserved for intra-operator clustering only.

## Onboarding & recovery

**Invite redemption flow**
The end-to-end path a new user follows from clicking an invite link to landing in their first chat room. Ordered:

1. Client deep-link handler receives the invite link.
2. Client checks for an existing DID on the device.
3. If no DID: client offers "Create new identity" or "Recover existing identity." Create generates a keypair + mnemonic; recover prompts for a mnemonic and rebuilds keys + bundle key.
4. **Mnemonic gate** (create path only): full 12-word mnemonic displayed; user must check "I have written these down" to proceed. No skip button.
5. Client redeems the invite by posting `(invite_token, did, public_key, sender_signature)` to the anchor (for anchor invites) or to the host server (for room invites).
6. Anchor / host server creates the user record and returns the first session token + refresh token + empty PPE + empty private blob (or the room membership for room invites).
7. Client prompts (skippable) for display name + avatar.
8. Client lands the user in the invited room and renders the server's channel list.

**Mnemonic gate**
The screen during create-new-identity that displays the user's 12-word recovery phrase and requires a checkbox acknowledgement ("I have written these down") before the invite can be redeemed. v1 uses a single checkbox; verification-by-re-entry (asking the user to retype 3 of the 12 words) is a future hardening step. Wording must make clear that Yawp cannot reset the phrase: "These 12 words are the only way to recover your account if you lose your device. Yawp cannot reset them."

**Second-anchor nudge**
A one-time prompt fired ~7 days after first signup, encouraging the user to add a second anchor for redundancy. Reinforces the "minimum 1, recommend 2+" anchor policy. Dismissable; not re-prompted in v1.

**Invite link**
A signup token issued by an anchor operator. Two flavors:

- _Single-use, time-limited_: granted to a specific recipient, valid for one signup, expires after a configurable window.
- _Shareable link_: anyone with the link may sign up, optionally with an expiry.

Invite links carry the anchor hostname; the anchor named in the link becomes the new user's first anchor.

**Mnemonic / Recovery phrase**
A BIP-39 12-word phrase shown to the user on first launch. The phrase encodes a master seed from which the user's identity keypair AND a bundle-encryption key are derived (HKDF). Losing the mnemonic without access to a paired device or anchor backup = losing the identity.

**Key bundle**
The encrypted blob held at the user's anchors containing device subkeys, anchor list, room memberships pointer, and other state needed for full recovery. Encrypted with a key derived from the user's mnemonic seed. Anchors store ciphertext only.

**Device pairing**
The QR-code / short-code flow for adding a new device without typing the mnemonic. An existing device signs a delegation that authorizes the new device's subkey. Preferred multi-device path; mnemonic is the cold-recovery fallback.

## Message lifecycle

**Message edit**
A signed append-only event that supersedes the body of a prior message. Original and every edit are retained. UI renders the most recent edit by default with an optional "view edit history" affordance. Each edit is signed by the original sender's device subkey.

**Message tombstone**
A signed delete event that wipes the body of a prior message while preserving its position in the timeline. UI renders the slot as "[deleted]." Body is removed from the database; the tombstone records who deleted (sender or moderator) and when. Default behavior — the body is unrecoverable.

**Body archive (opt-in)**
A server-level setting under server setup that, when enabled, causes deletes to additionally archive the original body to an admin-only store rather than wiping it. The tombstone still hides the body from the channel timeline; only privileged operators can retrieve the archived bodies via admin tooling. Off by default. Intended for compliance or leak-investigation use cases.

**Retention policy**
A per-channel (with per-server default) setting controlling automatic tombstoning of messages older than a cutoff. Values: `forever` (default) or a duration (e.g., 1d / 7d / 30d / 90d / 1y / custom). A daily background job tombstones messages past the cutoff. Independent of manual deletes.

**History-on-join**
Whether a freshly-joined member can read messages sent before their join event. Governed by a permission bit (`read_history_before_join`) on the role/override matrix. For plaintext rooms, default-on for `Member`+ on server-public channels; configurable per channel. For E2EE rooms (M10+), the bit is ignored — new joiners can only see messages from their join time forward.

**Server-authoritative ordering**
Message ordering, edit ordering, and tombstone ordering are determined entirely by the host server's monotonic insertion serial. Client-supplied timestamps are used only for optimistic-render correlation; the canonical "sent at" displayed in the UI is the server's insert timestamp. This applies per-room — DM ordering retains its own multi-anchor protocol.

## Message structure

**Mention**
A structured token in a message body that targets one or more users for notification. Four kinds in v1: `@user` (one DID), `@role` (all members holding a server role), `@here` (all currently-online members of the channel), `@everyone` (all members of the channel). Each kind is permission-gated on the broadcast forms; `@role`/`@here`/`@everyone` require the `mention_everyone` / `mention_role` permission bits (default: Owner + Admin only). Messages carry both a plaintext rendering and a `mentions: [{type, target}]` array for notification dispatch and client re-rendering.

**Reply**
A regular message that visibly references a single prior message via a `reply_to_message_id` field. Renders the quoted message as a card above the new one. By default, replying mentions the original author (one of the standard `@user` mention kinds); the sender may opt out per-reply.

**Thread**
A separate conversation branch anchored to a message in a parent channel (Discord/Slack style). **Not in v1** — deferred to post-M7. The message-row schema is forward-compatible: when threads ship, they will be identified by a shared `thread_root_id` derived from the message they anchor on, and the existing `reply_to_message_id` field handles inline references in the meantime.

**Reaction**
A small `(message_id, user_did, emoji)` event recording a user's tap-to-react on a specific message. Signed by the reacting user. v1 supports Unicode emoji only; custom server emoji deferred. Permission-gated on the `add_reactions` bit, default-on for `Member`+.

**Attachment**
A file uploaded alongside a message. Identified by an `upload_id` and a SHA-256 `content_hash`. The hash is included in the message envelope so the sender's signature covers the file's content (the host cannot silently substitute the file). Defaults: 25 MB per file, 10 attachments per message, no MIME allowlist, no virus scanning — all operator-configurable.

**Attachment backend**
The operator-chosen storage destination for attachment bytes. Two backends in v1: `local` (host server's disk, default) and `s3` (any S3-compatible object store — AWS S3, Cloudflare R2, MinIO, Backblaze B2). Clients see an identical wire format regardless of backend; the host returns a signed time-limited URL for download. Configurable per server.

**Encrypted attachment**
The M10+ equivalent for E2EE rooms: file is encrypted client-side with a per-file random symmetric key before upload; ciphertext is stored on the host; the symmetric key and content hash travel inside the encrypted message envelope. Host serves bytes but cannot read them. Decryption and hash verification happen on the recipient device.

## Notifications & unread

**Read marker**
The `(user, channel, last_read_message_id)` tuple recording how far a user has read in a given channel. Stored at the host server in v1 (plaintext rooms) — single round-trip update from any device, push-propagated to the user's other connected devices via that host's Phoenix Channel. DM read markers live at the recipient's anchor and propagate via the always-on anchor websocket. (E2EE rooms in M10 will revisit to client-derived markers.)

**Notification level**
A per-conversation setting controlling whether a message produces a push / banner notification. Three values: `all` (every message notifies), `mentions_only` (only @-mentions of the user, role, @here, @everyone), `muted` (no notification; unread badge still rendered). Applies to channels and DMs alike.

**Notification policy**
The user's full preference map: per-server defaults, per-channel overrides, per-DM overrides. Canonical copy lives in the user's private blob and syncs via anchors. A **mirror copy** is pushed to each host server that holds channels for the user, so the host can decide whether to fire a push without contacting the user's anchor on every message.

**Default notification level**
The v1 baseline applied when a user joins a new conversation without setting an explicit preference: `mentions_only` for channels, `all` for DMs. Reflects the asymmetry between high-intent (DM) and broadcast (channel) communication.

**Notification envelope**
A signed federation message constructed by a host server when one of its channels triggers a push for a user anchored elsewhere. Contents: `{user_did, source: "room_message" | "room_mention" | "dm", source_server, room_id_or_thread_id, message_id, timestamp}`. **No message body** — the device fetches the actual content from the source server after the push wakes it. Signed by the source server's keypair; verified by the recipient anchor before forwarding to the user's devices. Subject to Tier 2 per-anchor delivery budgets.

**Push fan-in**
The routing pattern by which all push notifications for a user flow through the user's anchor servers, not directly from source servers to mobile push services. Guest servers send notification envelopes to the user's anchor; the anchor decides per-device whether to push via the always-on websocket or APNs/FCM using the device push registry. Single fan-in point per user — guest servers never hold APNs/FCM credentials for users they don't anchor.

## Provisioning & administration

**Operator** (in the provisioning sense)
The human who runs the VPS — the person with shell access to the box where Yawp is deployed. **Distinct from any chat identity**: an operator does not necessarily have a DID. The operator authenticates against the admin panel with traditional credentials (password / magic link / passkey) set up at first boot; they do not sign messages, do not participate in federation, and do not appear in any chat roster _qua operator_. If the operator also wants to be the chat owner of their server, they do so by claiming the server with a DID — but that's a separate, optional step.

**Admin panel**
A Phoenix LiveView interface at `/admin`, password / passkey gated, for operator-level configuration that does not belong in the chat client: attachment backend selection and credentials, TURN/coturn configuration, per-server defaults (retention, attachment size limits, voice participant cap), the body-archive toggle, key rotation tooling, federation status, operator audit logs, and **chat-owner management** (generating claim tokens, viewing current chat owner, revoking and re-issuing claim tokens to transfer chat ownership). The admin panel is the operator's universe; the chat client is the chat owner's universe. The two are deliberately independent.

**Admin account**
A traditional credentials-based account that authenticates against the admin panel. Created during first-boot setup by visiting the URL printed in the server's startup logs. Has nothing to do with DIDs, keypairs, or federation. Multiple admin accounts may exist on a single server if the operator wishes (e.g., for sysadmin handoff).

**Claim token**
A short-lived one-time token generated **from inside the admin panel** by an authenticated operator and copied to the operator's clipboard. Pasted into the Yawp client when adding the server, where it binds the claiming user's DID to the server's `Owner` chat role. Invalidated immediately on successful claim. May be revoked from the admin panel before consumption.

**Server claim**
The flow that assigns a chat owner to a server. Operator generates a claim token in the admin panel → operator opens the Yawp client, adds the server, pastes the claim token → client posts a signed claim request `(claim_token, did, public_key, sender_signature)` to the server → server verifies, records the claiming DID as `Owner`, consumes the token. The admin panel afterward displays the chat owner's DID alongside the admin account.

**First-boot seeding**
Defaults populated automatically when a server boots for the first time, before any claim: the server's federation keypair, the three system roles, and two default channels (`#general` text, `General` voice). The seeded `Owner` role has no assignee until the chat-claim flow completes. The admin panel may exist (or be set up) independently of whether the chat owner has been claimed yet.

## Search & local storage

**Server-side search**
Full-text search performed by the host (for room messages) or the user's anchor (for DMs). v1 implementation: Postgres `tsvector` columns with GIN indexes on message body, author DID, channel ID, timestamp, and mentions. Permission-filtered — queries return only hits in conversations the searcher has `read_messages` permission for. Scoped to one server at a time; cross-server search is deferred.

**Client-side search**
Full-text search performed locally on the device using a local index built over decrypted message content. Mandatory in M10+ for E2EE rooms and E2EE DMs (server cannot read ciphertext, so server-side search is impossible). Results are limited to messages the device has actually received and stored.

**Local message store**
A persistent on-device store holding decrypted messages (and decrypted attachment metadata) for every conversation the user is a member of — DMs and channels. Required by M10+ to enable client-side search and offline reading of E2EE content. Storage budget and eviction policy are client decisions (no protocol commitment), but the store must hold at minimum the conversation tail and any historical ranges the user has explicitly fetched. Web clients use IndexedDB; native clients use platform-native storage (SQLite/Realm).

## Room encryption

**Plaintext room**
A room whose messages are stored as plaintext on the host server. The host operator can read all message content. TLS protects messages in transit; sender-signature on each message protects against host forgery, but not against host reading. The default and only available mode in v1 (pre-M10).

**E2EE room**
A room whose messages are end-to-end encrypted to each member's device subkeys. The host stores ciphertext only and cannot read message content. Introduced in M10. New rooms are E2EE-by-default after M10 ships. A room is **either plaintext or E2EE for its entire life** — there is no conversion in either direction.

**Danger zone**
The settings region where users may opt _out_ of secure defaults. Post-M10, creating a plaintext room is a Danger zone action: the UI requires an explicit acknowledgement that the server operator will be able to read all messages. Reserved for operator-managed use cases that need server-side search, history export, or compliance logging.

## Abuse & moderation

**Message request**
A first-time DM from a sender who is neither in the recipient's accepted contact list nor in a shared room with the recipient. Held in a separate "Message Requests" inbox until the recipient explicitly accepts. After acceptance, future DMs from that sender bypass the request queue.

**Per-anchor delivery budget**
A rate limit enforced by the receiving anchor on how many deliveries it accepts from each peer anchor per unit time. Default starting budgets are low (100/min) and scale up as a peer accumulates a history of legitimate, accepted traffic. Defense against open-federation spam.

**Operator block**
A unilateral decision by an anchor operator to refuse all federation traffic from a named peer anchor. The blocked anchor's users see a clear "blocked by recipient anchor" error on attempted deliveries so they can re-anchor elsewhere.

**Banning a user**
The act by which an anchor operator removes a user from their server. Banning is atomic: it removes the user from all rooms hosted on that server AND, if the server was anchoring for that user, terminates the anchor relationship. There is no "remove anchor without banning" operation. Anchor service and room presence rise and fall together.

## Bots & automation

_Proposed, post-v1. See [ADR 028](docs/adr/028-bot-scripting-and-interaction-model.md)._

**Bot**
An automation participant that is a first-class **identity** — its own DID and keypair, like any user. A bot is not a privileged side-channel of the server; its authority in a room derives entirely from a constrained server role (a permission bit-set) and per-channel overrides, exactly as a human's does. Bots run user-authored automation logic on the server that hosts them.

**Command**
A bot action a user **explicitly invokes**, with arguments (e.g. `/weather Toronto`). The arguments are consensual input handed directly to the bot — not the bot observing the room. Commands are the primary way a user drives a bot.

**Widget**
An interactive component a bot renders into a room. A user interacting with a widget returns a **structured payload** to the bot. Together with commands, widgets are the bot's entire input surface. The widget set is a **closed, versioned catalogue** — not an open markup/HTML surface — so every widget renders identically on web and native and stays sandbox- and E2EE-safe. v1 catalogue: **button / quick-reply** (tap returns an action id; RCS-style suggested actions), **card** (title, body, image reference, and a row of buttons), **form** (ordered fields — text, number, single-select, multi-select; submit returns a keyed payload), and **poll/survey** (a form specialization with aggregate result rendering). Deferred: free-form layouts, embedded media playback, live-updating/in-place widgets, date pickers, file inputs, multi-step wizards.

**Interaction-scoped bot** (default)
A bot that **cannot read room content** — neither history nor the live message stream. Its only inputs are the commands invoked on it and the widget interactions addressed to it; each is delivered to the bot's identity separately from the room's message envelope. Interaction-scoped bots work in every room, encrypted or not, and uphold the property "bots cannot read your conversations." This is the default and only kind unless a room is explicitly configured otherwise.

**Keyed bot** (opt-in)
A bot that has been **explicitly granted a room's content** by being added as a keyed participant — issued the room's encryption key like any member. This is the only path to ambient/listener automation (auto-moderation, conversation translation, full logging). Never the default; must be surfaced explicitly to the room's members because it changes the room's privacy properties; a room may forbid keyed bots entirely as policy.
